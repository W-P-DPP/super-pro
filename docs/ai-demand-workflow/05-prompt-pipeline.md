# 05 提示词流水线

## 目标

提示词流水线负责把文本集合转成结构化问题、机会摘要、评分结果和 PRD 草案。每个阶段都必须有稳定输入、清晰输出和明确失败处理。

## 设计规则

- 一个提示词只承担一个职责
- 优先输出结构化结果，建议使用 JSON Schema
- 必须记录提示词名称、版本、模型、输入快照和输出快照
- 更换模型不应改变业务状态机
- 评分维度必须来自 `scoring_profile`，不能把固定商业假设写入提示词模板

## 阶段 1：信号归一化

输入：

- 原始标题
- 原始正文
- 来源元数据

输出：

- 归一化文本
- 简要问题摘要
- 用户角色提示
- 噪声或垃圾判断
- 已脱敏内容

规则：

- 保留原始含义
- 显式区分事实和推断

## 阶段 2：去重与合并建议

输入：

- 单条 `cleaned_signal` 或一小组候选信号

输出：

- 重复标记
- 候选重复目标
- 去重理由

规则：

- 只给建议，不做破坏性删除
- 低置信度结果进入规则校验或人工队列

## 阶段 3：问题提取与聚类标注

输入：

- 一组 `cleaned_signal`

输出：

- 聚类标题
- 聚类问题陈述
- 共性痛点
- 代表信号 ID
- 不应合并的边界说明

规则：

- 以用户问题相似度聚类，而不是只看表层措辞
- 每个聚类都必须能用一句话解释

## 阶段 4：机会摘要生成

输入：

- `problem_cluster`
- 代表信号
- 来源上下文

输出：

- `opportunity_brief`
- 证据摘要
- 目标用户
- 分发假设
- 结果可见性假设
- 证据缺口说明

规则：

- 必须显式指出当前证据不支持的结论
- `opportunity_brief` 是评分和 PRD 生成的统一上游对象

## 阶段 5：机会评分

输入：

- `opportunity_brief`
- `scoring_profile`
- 当前策略上下文

输出：

- 分项分数
- 总分
- 评分理由
- 关键风险
- 待人工验证问题

规则：

- 每个分项都必须附解释
- 评分维度取自 `scoring_profile`
- 默认建议维度可包含需求强度、战略匹配度、实现可行性、流量或分发潜力、结果可见性、证据置信度
- 商业化潜力只作为可选维度

## 阶段 6：PRD 前门禁辅助

输入：

- `opportunity_brief`
- 评分结果
- 门禁规则

输出：

- 门禁建议
- 缺失证据清单
- 是否建议进入 PRD 生成

规则：

- 这是辅助判断，不替代人工 Gate 1 决策
- 如果证据不足，应优先返回补证据而不是强行生成 PRD

## 阶段 7：PRD 草案生成

输入：

- 已通过 Gate 1 的 `opportunity`
- `opportunity_brief`
- 证据集合

输出：

- 结构化 `prd_draft`
- 章节级引用
- 待验证假设

规则：

- 未验证内容不能写成已确认需求
- 范围内和范围外必须明确

## 阶段 8：PRD 后审查辅助

输入：

- `prd_draft`
- 审查意见
- 版本历史

输出：

- 修改建议
- 冲突摘要
- 缺失证据清单

## 输出契约

每个提示词输出至少包含：

- `result`
- `confidence`
- `evidence_ids`
- `assumptions`
- `warnings`

## 固定 prompt_stage 标识

为避免后续服务和审计表使用不同命名，建议固定使用以下 `prompt_stage` 值：

- `signal_normalization`
- `signal_deduplication`
- `problem_clustering`
- `opportunity_brief_generation`
- `opportunity_scoring`
- `pre_prd_gate_assist`
- `prd_draft_generation`
- `post_prd_review_assist`

## 失败处理

- JSON 解析失败：标记失败并按上限重试
- 低置信度：进入人工队列
- 证据不足：停止下游生成，回到 `opportunity_brief` 补证据
- 门禁未通过：保留记录，不删除已有评分或草案版本

## 评估指标

- 结构化输出成功率
- 空输出或低质量输出率
- Gate 1 退回率及退回原因清晰度
- Gate 2 首轮“可审查”通过率
- 同一输入重跑的一致性
