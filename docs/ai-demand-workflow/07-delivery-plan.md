# 07 交付计划

## 交付目标

交付一个最小可运行实现：能把公开需求信号推进到 `opportunity_brief`、可配置评分、PRD 草案和双门禁审查，同时给未来接入 super-pro 留出清晰路径。

## 一期北极星

一期不是追求“自动生成更多 PRD”，而是追求“稳定地产出可审查、可追溯、可被人工决策的 PRD 候选项”。

## 一期成功指标

- `100%` 的 PRD 草案都来自已通过 Gate 1 的机会
- `100%` 的最终结论都能追溯到证据、提示词执行和人工决策
- `>= 80%` 的成熟聚类能形成 `opportunity_brief`
- `>= 70%` 的首版 PRD 草案被评为“可审查”

## 阶段 1：文档与契约

交付物：

- 当前这套文档
- 核心状态枚举
- DTO 和 Schema 草案
- 提示词输出 Schema 草案
- `opportunity_brief` 与 `scoring_profile` 契约
- 数据库模型设计稿 `10-database-model-design.md`

完成标准：

- 产品、研发、AI 负责人对对象模型、双门禁和评分配置方式达成一致
- 数据落库边界、审计链路和 migration 顺序已明确

## 阶段 2：数据与采集基础

交付物：

- `source_config`、`collection_batch`、`raw_signal`、`audit_event` 基础表
- 手工导入接口
- 一个最小可用的定时采集器

完成标准：

- 公开信号能稳定入库，并具备批次级追踪能力

## 阶段 3：清洗、聚类与机会摘要

交付物：

- 清洗流水线
- 去重和噪声标记
- `problem_cluster` 相关表和任务
- `opportunity_brief` 生成任务

完成标准：

- 一个信号批次可以产出可读的聚类和可审查的 `opportunity_brief`

## 阶段 4：评分与 PRD 前门禁

交付物：

- `scoring_profile` 配置能力
- 机会评分流程
- Gate 1 审查流
- 评分与门禁审计记录

完成标准：

- 一个 `opportunity_brief` 能按指定策略完成评分，并得到明确的 Gate 1 结论

## 阶段 5：PRD 生成、PRD 后门禁与运营闭环

交付物：

- PRD 生成流程
- Gate 2 审查流
- 状态流转 API
- 最小审计查询页或查询 API

完成标准：

- 产品经理能完成一次从证据到 PRD 再到审查结论的完整闭环

## 建议时间线

- 第 1 周：文档、契约、状态机、评分配置
- 第 2 周：采集与原始存储
- 第 3 周：清洗、去重、聚类、`opportunity_brief`
- 第 4 周：评分、Gate 1、提示词审计
- 第 5 周：PRD、Gate 2、审计查询、验收检查

## 主要风险

- 来源质量不足会拖低聚类和机会摘要质量
- 提示词输出波动会影响结构化稳定性
- 评分策略不透明会削弱审查者信任
- 门禁规则不清会导致 PRD 被过早生成
- 审计记录缺失会阻断后续回放和排障

## 风险缓解

- 从少量高质量公开来源开始
- 每个提示词阶段都强制结构化输出和有界重试
- 从第一版开始就启用 Gate 1、Gate 2 和人工覆盖
- 优先落地 `prompt_execution`、`state_transition`、`review_task`
- 将评分维度和权重放入 `scoring_profile`，避免硬编码策略
