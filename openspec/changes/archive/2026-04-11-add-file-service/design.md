## Context

当前后端采用清晰的业务域分层结构，每个业务域都按 `entity/controller/dto/repository/router/service` 组织。仓库根目录使用 `pnpm workspace + turbo` 管理多个 package，但尚未存在面向本地文件管理的业务模块和前端工程。

本次需求非常明确：

- 仓库根目录存在固定的 `file` 目录，作为用户可发现、可查询、可上传、可创建子目录的根目录
- 仓库根目录存在固定的 `rubbish` 目录，作为删除后的回收区
- 后端需要提供最小可用的文件服务能力
- 前端需要新增独立工程 `file-server`，由 monorepo 统一管理

这不是对象存储、权限中心或网盘系统的设计问题，而是一个基于仓库本地文件系统的受控文件管理能力。

## Goals / Non-Goals

**Goals:**

- 新增 `file` 后端业务模块，并保持现有后端分层风格
- 仅允许在仓库根目录 `file` 内创建文件夹、上传文件、查询文件树
- 删除文件或文件夹时，将目标移动到 `rubbish`，而不是物理删除
- 在 `rubbish` 中保留原相对路径，并按删除时间分桶
- 上传同名文件时拒绝覆盖
- 新增独立前端工程 `file-server` 并纳入 monorepo

**Non-Goals:**

- 本次不引入数据库文件元数据表
- 本次不实现文件恢复接口
- 本次不实现文件重命名、移动、复制、搜索、预览、下载鉴权等扩展能力
- 本次不支持自动创建缺失的父级目录
- 本次不把 `rubbish` 暴露为用户可直接操作的业务目录

## Decisions

### 决策 1：`v1` 不引入数据库，文件系统即真实数据源

本次能力的真实状态完全体现在磁盘目录结构中：

- 查询依赖扫描 `file` 目录
- 创建文件夹依赖文件系统目录操作
- 上传依赖文件写入
- 删除依赖目录移动到 `rubbish`

在这种场景下，如果额外写入数据库，会立即引入双写一致性、补偿、启动同步和异常恢复问题，但并不能带来当前阶段必要的业务价值。因此 `v1` 直接以文件系统为唯一真实数据源。

后续只有在出现以下需求时，再考虑引入数据库：

- 文件 owner、权限、审计
- 标签和业务关联
- 回收站恢复记录
- 搜索索引、去重或哈希

### 决策 2：使用仓库根目录固定根路径 `file` 与 `rubbish`

后端以仓库根目录为基准，固定解析两个业务目录：

- `file/`: 可操作根目录
- `rubbish/`: 软删除归档目录

所有业务路径都使用“相对 `file` 根目录”的表达方式，例如 `/docs/a.txt`。系统在执行任何操作前都先将相对路径解析为绝对路径，再进行合法性校验。

这样做优于允许外部传入任意绝对路径的原因：

- 边界明确，易于保护
- 前后端协议更简单
- 能从根本上避免误操作仓库中其他目录

### 决策 3：所有路径操作都必须通过规范化和根路径校验

本次实现的核心风险不是 CRUD 本身，而是路径逃逸。后端必须对所有输入路径执行以下处理：

1. 规范化用户输入路径
2. 将路径解析为 `file` 根目录下的绝对路径
3. 校验解析后的最终路径仍然位于 `file` 根目录内
4. 只有校验通过后才允许继续执行

必须显式拒绝：

- `../` 等路径穿越
- 指向 `file` 根目录之外的路径
- 删除 `file` 根目录本身

### 决策 4：删除采用“移动到回收区”而不是物理删除

删除文件或文件夹时，不直接调用物理删除，而是将目标移动到：

```text
rubbish/<timestamp>/<original-relative-path>
```

例如删除 `/docs/report.pdf` 后，结果可能为：

```text
rubbish/2026-04-11_15-30-22/docs/report.pdf
```

这样设计的原因：

- 保留原目录结构，未来恢复更容易
- 通过时间分桶避免不同删除批次之间冲突
- 同名文件不会相互覆盖

### 决策 5：上传拒绝覆盖，目标父目录必须已存在

用户已明确要求上传同名文件时拒绝覆盖，因此上传接口采用以下约束：

- `targetPath` 必须指向已存在的目录
- 若目标目录下已存在同名文件或同名文件夹，则直接返回受控错误
- 不自动创建缺失父目录

同样的约束也适用于新建文件夹：

- 目标父目录必须存在
- 同级目录下若存在同名文件或文件夹，则拒绝创建

这样做虽然少了一些“自动帮用户修正”的便利，但行为更可控，也更符合受限目录管理的预期。

### 决策 6：查询接口直接返回 `file` 根目录树结构

`GET /api/file/tree` 直接扫描 `file` 根目录并返回树结构，节点至少包含：

- `name`
- `relativePath`
- `type`，取值为 `file` 或 `folder`

文件节点建议额外包含：

- `size`
- `modifiedTime`

目录节点建议包含：

- `children`

该接口只反映 `file` 目录，不返回 `rubbish` 内容。这样前端视图天然与受控业务空间保持一致。

### 决策 7：新增独立前端工程 `file-server` 并纳入 monorepo

前端不复用现有模板页面，而是新增一个独立 package：`file-server`。该工程通过现有 workspace 和 turbo 配置接入仓库，作为文件管理的专用入口。

`v1` 前端页面可保持极简：

```text
┌───────────────────────────────────────┐
│ 顶部：上传 | 新建文件夹 | 删除        │
├───────────────┬───────────────────────┤
│ 左侧：目录树   │ 右侧：当前目录内容     │
│               │ 名称 大小 更新时间     │
└───────────────┴───────────────────────┘
```

这已经足以承接后端接口，不需要一开始扩展成复杂网盘系统。

## Risks / Trade-offs

- [风险] `GET /api/file/tree` 在文件量很大时会有递归扫描成本
  -> Mitigation：`v1` 先以完整树返回满足需求，后续若目录规模变大，再拆分懒加载或分页接口

- [风险] 大目录删除本质是移动目录，仍可能有较重 IO 成本
  -> Mitigation：`v1` 保持同步移动实现，后续如有需要再增加异步任务化处理

- [风险] 不引入数据库意味着没有天然审计和检索能力
  -> Mitigation：当前需求聚焦本地受控目录管理，等出现权限、搜索、恢复等需求时再补数据库

- [风险] 用户可能误以为删除后可以直接在前端恢复
  -> Mitigation：在接口和前端文案中明确“删除将移入回收目录，当前版本不提供恢复接口”

## Migration Plan

1. 新增 OpenSpec 变更文档和能力 spec
2. 在后端新增 `file` 业务模块，定义 DTO、controller、service、repository、router
3. 实现根路径解析、路径越界保护、目录树扫描、文件夹创建、文件上传、软删除移动
4. 在 `general-server/src/index.ts` 中挂载 `/file` 业务路由
5. 补充单元测试和集成测试，覆盖成功与错误分支
6. 新增 `file-server` 前端工程，并接入 `pnpm-workspace.yaml`、根 `package.json`、`turbo.json`
7. 实现最小可用的文件管理页面并联调后端接口

回滚策略：

- 回滚后端 `file` 模块与路由挂载
- 回滚 `file-server` package 与 workspace 配置
- 仓库根目录中的 `file/` 与 `rubbish/` 作为业务数据目录保留，不自动删除其中内容

## Open Questions

- 上传接口是否只支持单文件上传；`v1` 建议先限制为单文件
- 删除接口是否允许批量删除；`v1` 建议先限制为单目标删除
- 是否需要在前端提供上传进度；`v1` 可暂不提供
