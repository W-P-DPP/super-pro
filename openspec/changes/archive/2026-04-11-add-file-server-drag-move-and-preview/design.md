## Context

当前 `file-server` 已经具备树形浏览、创建目录、上传和删除能力，但树节点仍是静态操作入口，用户无法直接通过拖动调整目录结构。同时右侧内容区仍为占位态，导致用户查看文件内容时需要离开当前工作区，破坏“左树管理 + 右侧查看”的基本工作流。

这次变更同时跨越前端树交互、右侧内容区、`general-server/src/file` 后端接口与文件访问策略，属于典型的跨模块能力扩展。现有约束仍然成立：

- 所有文件操作必须限制在仓库根 `file/` 目录内
- 不允许覆盖目标文件或目录
- 文件服务返回受控中文反馈
- `file-server` 需要保持树为主工作区，右侧只承接与当前选中节点相关的辅助能力

## Goals / Non-Goals

**Goals:**

- 支持在树中拖动文件或文件夹，并将其移动到其他文件夹下
- 为拖动移动提供后端受控移动能力，拒绝越界、移动到自身/子孙目录、目标冲突和非法目标
- 将右侧内容区升级为预览区，和左树当前选中节点联动
- 支持常见文件的受控预览，包括 `md`、`pdf`、图片、音视频、`docx`、`xlsx/xls` 等
- 为加载中、预览失败、不支持预览和超大文件回退提供统一反馈

**Non-Goals:**

- 不实现树节点同级排序或拖动后自定义插入位置
- 不实现跨业务域、跨根目录或跨系统存储的移动
- 不实现在线编辑、批注、保存回写或版本管理
- 不保证所有二进制格式都能原样高保真预览；对不支持格式允许受控降级

## Decisions

### 决策 1：拖动语义限定为“移动到目标文件夹下”

树拖动只允许把文件或文件夹放入某个文件夹节点，不支持放到文件节点，也不支持用拖动表达同级重排。这样可以把交互语义和后端契约都收敛为单一的“move target to destination folder”。

选择这个方案的原因：

- 后端只需处理目录移动，不需要额外维护顺序模型
- 与现有文件系统语义一致，用户理解成本低
- 可以直接复用现有受控根目录与冲突校验模式

备选方案：

- 支持树节点重排与插入位置：交互更复杂，但当前文件系统并无稳定顺序语义，收益低于成本

### 决策 2：后端新增单一受控移动接口，由 repository 统一做路径与冲突校验

后端在 `file` 业务域内新增移动接口，例如 `POST /api/file/move`，输入至少包含 `sourcePath` 与 `destinationPath`。`service` 负责输入规整和业务错误映射，`repository` 负责：

- 根目录内解析
- 目标目录存在且为文件夹
- 源节点存在且不能为根目录
- 禁止移动到自身或其后代目录
- 拒绝目标目录中同名冲突

选择这个方案的原因：

- 延续当前 `file` 模块的分层约束
- 拖动移动与未来非拖动移动入口可以共用同一能力
- 可直接复用已有中文错误与测试结构

备选方案：

- 前端通过删除 + 重传模拟移动：无法保留原文件内容与目录结构的一致性，也不满足同盘移动语义

### 决策 3：右侧内容区升级为“按类型分层渲染”的预览区

右侧内容区不再是纯占位区，而是根据当前选中文件决定预览策略：

- 文本/Markdown：读取文本内容并在前端渲染，Markdown 走受控解析
- 图片、音视频、PDF：优先使用浏览器原生能力预览
- `docx`、`xlsx/xls`：使用浏览器侧解析或嵌入式渲染方案生成只读预览
- 不支持或过大的文件：展示受控提示，不强行预览

选择这个方案的原因：

- 保留“左树管理，右侧查看”的清晰分工
- 不要求所有格式都走同一技术路径，降低实现耦合
- 能逐步扩展支持类型，而不破坏主交互

备选方案：

- 统一依赖外部在线预览服务：实现快，但引入外部依赖、隐私风险和网络不确定性
- 统一后端转码：服务复杂度和资源消耗过高，不适合当前仓库型文件服务

### 决策 4：预览内容通过受控文件读取能力提供，避免直接暴露仓库根路径

预览需要稳定访问文件内容，但不能把仓库 `file/` 根目录直接裸暴露给前端。因此实现上应通过 `file` 业务域提供受控读取路径或流式内容接口，并继续复用根目录校验。

这样做的原因：

- 预览链路与文件管理链路共享同一套安全边界
- 可以按类型或大小做预览准入控制
- 为后续下载、只读查看、缓存头控制保留扩展空间

备选方案：

- 直接把 `file/` 挂成公开静态目录：实现简单，但会绕过现有业务边界与受控校验

## Risks / Trade-offs

- [办公文档预览在浏览器侧解析成本较高] → 对超大文档设置受控降级，不阻塞基础图片/PDF/文本预览
- [拖动交互容易出现误操作] → 增加明确的拖动态、目标高亮和失败反馈，禁止无效投放目标
- [目录移动存在循环引用风险] → repository 在最终落盘前显式拒绝移动到自身或后代目录
- [预览能力会增加前端依赖体积] → 仅按需引入解析能力，并优先选择可懒加载方案
- [二进制预览的格式覆盖不可能一步到位] → 明确“不支持预览”的受控状态，而不是返回空白区

## Migration Plan

- 先增加后端移动与预览读取能力，并补齐单元/集成测试
- 再接入前端树拖动状态与右侧预览区
- 最后补充支持格式、加载态和失败态验证
- 回滚时可先下线前端拖动与预览入口；后端新增接口保留但不被前端调用，不影响现有上传/删除/建目录能力

## Open Questions

- `docx`、`xlsx/xls` 的首版预览是优先使用浏览器侧轻量解析，还是引入更完整但更重的预览依赖
- 预览文件大小阈值应如何定义，才能在可用性与浏览器负载之间平衡
- 是否需要在首版同时提供“下载原文件”作为预览失败或不支持时的标准回退动作
