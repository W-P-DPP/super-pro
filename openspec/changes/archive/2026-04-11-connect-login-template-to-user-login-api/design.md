## Context

`login-template` 当前是独立的 Vite + React 登录模板，页面已经具备较完整的视觉外壳、明暗主题切换和基础表单交互，但认证流程仍依赖 `src/lib/mock-auth.ts` 中的本地模拟逻辑。后端侧已经提供 `POST /api/user/loginUser`，返回统一的 `code/msg/data` 结构，其中 `data` 包含 `token`、`tokenType`、`expiresIn` 和脱敏后的 `user` 信息。

本次变更横跨登录页组件、前端认证工具、运行时配置和本地存储策略，且需要兼顾现有设计约束：
- 继续沿用现有 `shadcn` 风格与 token，不引入平行视觉系统
- 所有新增文案默认使用简体中文
- 登录、错误、加载和禁用态需要在浅色与深色主题下保持一致体验
- 当前模板没有路由与完整业务页，因此登录成功后的落点必须在现有单页结构内完成

## Goals / Non-Goals

**Goals:**

- 将登录表单接入真实接口 `POST /api/user/loginUser`
- 引入统一的认证请求与响应解析层，兼容后端 `code/msg/data` 包装
- 根据“记住当前设备”选项持久化登录态，并保存 token、过期时间和当前用户信息
- 调整登录页交互，使其与现有后端能力一致，提供可控的加载、成功和失败反馈
- 保持现有登录模板的主题、排版和组件语言不变

**Non-Goals:**

- 不在本次变更中新增注册接口、忘记密码、退出登录或刷新 token 能力
- 不在本次变更中引入全局状态管理库、请求库或路由系统
- 不在本次变更中设计新的业务首页或登录后跳转页面
- 不修改 `general-server` 登录接口契约

## Decisions

### 决策 1：新增轻量认证客户端模块，统一封装登录请求与响应解析

在 `login-template/src/lib/` 下新增认证相关模块，负责：
- 读取 `import.meta.env.VITE_API_BASE_URL`
- 发起 `POST /api/user/loginUser`
- 解析后端统一返回结构
- 将业务失败、网络失败和结构异常转换为页面可消费的中文错误

这样做优于把 `fetch` 和解析逻辑散落在 `LoginPage.tsx` 中，原因是：
- 页面组件只关心交互和状态，不直接承载接口协议细节
- 后续若新增“获取当前用户”或“退出登录”，可以复用同一认证客户端
- 可以集中处理后端统一响应包裹和错误兜底，减少页面重复判断

备选方案：
- 直接在页面内发请求：实现更快，但会把页面组件与接口协议强耦合，后续扩展成本更高
- 引入 axios/react-query：能力更强，但对当前单页模板而言超出必要范围

### 决策 2：登录成功后使用存储抽象保存认证态，按 remember 切换 `localStorage` 与 `sessionStorage`

新增认证存储工具，统一读写：
- `token`
- `tokenType`
- `expiresAt`
- `user`

当用户勾选“记住当前设备”时写入 `localStorage`；未勾选时写入 `sessionStorage`。应用启动时优先恢复 `localStorage`，其次恢复 `sessionStorage`。

这样做优于只使用单一存储介质或仅保存在内存中，原因是：
- 与当前界面已有“记住当前设备”交互语义一致
- 不引入复杂状态库也能覆盖刷新页面与会话级恢复场景
- 通过抽象层可避免页面组件直接操作多个存储 key

备选方案：
- 统一写入 `localStorage`：实现简单，但会让“记住当前设备”失去实际意义
- 仅保存到 React state：刷新后丢失登录态，不符合真实接入目标

### 决策 3：登录页收敛为真实登录能力，注册入口在本次变更中降级处理

由于后端当前仅提供登录接口，不提供注册接口，本次前端将不再保留可提交的注册 mock 流程。页面可采用以下之一：
- 直接移除注册模式切换
- 或保留视觉入口但置灰禁用，并明确提示“暂未开放”

实现时优先选择更简洁、与现有能力更一致的方案。

这样做优于继续保留本地 mock 注册，原因是：
- 避免页面同时呈现“真实登录 + 伪注册”的混合状态
- 降低联调歧义，避免用户误以为注册结果已落到后端
- 与当前后端能力边界一致，范围更清晰

备选方案：
- 继续保留本地 mock 注册：演示完整，但会制造错误预期

### 决策 4：登录成功后的可见结果以内嵌成功态和已登录信息展示为主，不新增跳转链路

当前模板没有业务首页与路由系统，因此登录成功后仍停留在现有页面结构中，但切换为“已登录”成功态，显示当前用户基础资料和登录结果。若后续有业务页，可在此基础上再接入跳转。

这样做优于临时新增一个占位跳转页，原因是：
- 不会为了接入接口额外扩展无业务价值的页面结构
- 保持本次范围集中在真实认证链路
- 更适合模板仓库当前的单页定位

备选方案：
- 新增成功页或伪仪表盘：视觉上更完整，但会把范围扩展到未定义的业务页面

### 决策 5：新增状态反馈必须复用现有主题 token 和组件视觉语言

成功、错误、加载、禁用与输入校验态继续复用当前登录页的边框、圆角、阴影和语义色，不新增绕过 token 的颜色和样式。

这样做优于为“真实接口接入”单独设计一套提示样式，原因是：
- 满足前端设计规范对统一视觉语言和低噪音交互的要求
- 保持浅色/深色主题下的体验一致
- 降低样式回归风险

## Risks / Trade-offs

- [风险] 后端接口基础地址在不同本地环境下可能不一致
  → Mitigation：使用 `VITE_API_BASE_URL` 管理，并在缺失配置时给出明确开发态提示

- [风险] 后端统一响应结构如果与预期不一致，页面可能出现“成功 HTTP 状态但无法登录”的异常
  → Mitigation：在认证客户端中增加响应结构校验和兜底错误文案

- [风险] 将注册入口下线后，现有模板演示能力会变窄
  → Mitigation：在 proposal 和任务中明确这是与后端能力对齐的范围收敛，不影响后续新增注册变更

- [风险] 登录成功后不跳转可能让用户感觉流程未完成
  → Mitigation：在已登录态中明确展示成功信息、当前用户信息和 token 已落地的状态反馈

## Migration Plan

1. 新增认证 API 与存储抽象，定义登录请求、响应和会话恢复工具
2. 改造 `LoginPage`，移除对 `mockLogin/mockRegister` 的依赖并接入真实登录提交
3. 调整页面交互，收敛注册入口，补齐加载、失败、成功和已登录态
4. 在应用入口恢复本地认证态，并验证浅色/深色主题下的展示一致性
5. 通过 `build`/`lint` 与联调检查后交付

回滚策略：
- 回滚 `login-template` 中新增的认证客户端与存储逻辑
- 恢复 `mock-auth.ts` 驱动的本地登录流程
- 保持后端接口不受影响

## Open Questions

- `VITE_API_BASE_URL` 是否需要在模板内提供默认开发地址，还是要求联调时显式配置？
