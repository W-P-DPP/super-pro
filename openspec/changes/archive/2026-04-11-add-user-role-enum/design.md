## Context

当前 `general-server/src/user` 已具备完整的 `entity/controller/dto/repository/router/service` 六层结构，并已经提供 `getUser`、`getUserDetail`、`createUser`、`updateUser`、`deleteUser` 等接口。但现有 `sys_user` 只有 `username`、`nickname`、`email`、`phone`、`status`、`passwordHash` 等字段，缺少可稳定表达用户身份类型的角色字段。

这次变更不是新增一个权限系统，而是在现有 `user` 域内把“管理员 / 访客”纳入正式数据模型，并同步到 CRUD 契约、校验逻辑和测试。由于仓库当前遵循中文响应、动作式路由和强分层约束，设计必须保持这些约定不变。

## Goals / Non-Goals

**Goals:**

- 为 `sys_user` 新增 `role` 字段，并在 `user` 实体与 DTO 中以受控枚举建模
- 让 `createUser`、`updateUser`、`getUser`、`getUserDetail` 的输入输出都能稳定处理 `role`
- 对角色字段提供统一的枚举校验和中文业务错误，禁止无效值透传到仓储层
- 为已有数据和旧调用方提供兼容行为，避免因为新增字段导致创建接口立即失效
- 补充与角色字段相关的单元测试和集成测试

**Non-Goals:**

- 不引入独立的角色表、权限表或 RBAC 关系模型
- 不在本次变更中新增按角色过滤、按角色授权或菜单权限控制能力
- 不重构现有 `user` 路由风格、响应包装结构或全局异常机制

## Decisions

### 决策 1：角色字段继续归属 `user` 域，不拆分独立角色模块

本次角色只有两个固定值：`admin` 和 `guest`，且用户需求仅要求在 `user` CRUD 中表达和维护该字段，不涉及复杂权限关系。因此角色字段直接放在 `sys_user.role` 中，并继续由 `user` 模块六层结构承载。

这样做优于新建角色模块的原因：

- 变更边界清晰，只扩展现有 `user` 域，不制造新的跨模块依赖
- 当前需求仅是基础身份枚举，不足以支撑独立角色模型的复杂度
- 与仓库现有“一个业务域维护一套实体和 CRUD 契约”的组织方式一致

备选方案：

- 新建 `role` / `auth` 模块并通过关联关系维护用户角色：扩展性更强，但对当前需求是过度设计

### 决策 2：角色字段使用字符串枚举，持久化值固定为 `admin` 和 `guest`

尽管需求中给出了“1：管理员；2：访客”的业务含义，本次后端字段仍采用字符串枚举值 `admin` 和 `guest` 作为对内对外的统一契约；数字序号仅作为文档中的枚举说明，不作为数据库和接口真实值。

这样做的原因：

- 当前 TypeScript、DTO 校验和接口响应更适合直接使用语义化枚举值，而不是额外维护数字到字符串的映射
- 可以减少前后端对“数字码值 + 展示值”的双重转换，降低实现和测试复杂度
- 与用户给出的枚举标识 `admin`、`guest` 保持一致，接口可读性更高

备选方案：

- 持久化为 `1` / `2` 并在服务层映射成字符串：兼容传统字典编码，但会增加 DTO、校验和响应映射复杂度

### 决策 3：创建用户时 `role` 缺省默认值为 `guest`

为避免新增必填字段直接破坏现有调用方，`createUser` 在未显式传入 `role` 时默认写入 `guest`；同时数据层也应保证已有记录在迁移后具备兼容默认值。

这样做的原因：

- 与“访客账号是系统最小权限身份”的业务语义一致
- 兼容旧的创建请求，降低接口升级成本
- 可以为已有历史数据提供统一兜底值，减少空值处理

备选方案：

- 将 `role` 设为创建必填：约束更强，但会立即破坏现有调用方并增加迁移成本

### 决策 4：角色校验集中在 Service 层，Repository 仅处理已规范化数据

`user.service.ts` 应负责把外部输入规范化为内部枚举，并在非法值出现时返回受控中文错误；`user.repository.ts` 只接收已经通过校验的 `role` 值并完成持久化。

这样做的原因：

- 保持 `controller -> service -> repository` 的职责边界清晰
- 可以在单元测试中直接覆盖业务规则，而不把校验逻辑散落到控制器或仓储
- 与仓库当前 `status`、`remark` 等字段的校验方式一致

备选方案：

- 在控制器层直接校验角色：实现简单，但会削弱业务规则复用性
- 在仓储层兜底校验：会让数据访问层承担不必要的业务逻辑

### 决策 5：角色字段纳入所有用户响应 DTO，保证 CRUD 返回结构一致

`getUser`、`getUserDetail`、`createUser`、`updateUser` 的返回 DTO 都应包含 `role`，避免部分接口有角色、部分接口没有角色，导致调用方需要额外兼容。

这样做的原因：

- 统一用户实体对外可见字段，减少前端和调用方的分支处理
- 与 CRUD 接口“新增字段后所有读操作可见”的常规契约一致
- 便于后续测试直接复用响应断言结构

备选方案：

- 仅在详情或创建响应中返回 `role`：改动更小，但契约不一致，容易引发前端遗漏

## Risks / Trade-offs

- [风险] 现有数据库和种子数据没有 `role` 字段，直接扩展实体后可能出现历史数据为空值
  Mitigation：在实体定义和初始化数据中提供默认值 `guest`，并在测试数据中同步补齐

- [风险] 采用字符串枚举与“1/2”文档描述存在表达差异，后续实现时可能被误解为数字字段
  Mitigation：在 spec 和 design 中明确“数字仅为枚举序号说明，实际契约值为 `admin` / `guest`”

- [风险] `UserResponseDto` 被其他接口复用后，新增 `role` 可能扩大返回面
  Mitigation：将这一变化视为兼容性增强，并在相关测试中显式断言新增字段存在

## Migration Plan

1. 扩展 `user.entity.ts`、`user.dto.ts` 和仓储输入输出类型，补齐 `role` 字段与默认值
2. 在 `user.service.ts` 中新增角色枚举定义、规范化函数和中文错误处理
3. 调整 `createUser`、`updateUser`、`getUser`、`getUserDetail` 的业务链路和响应映射
4. 更新单元测试与集成测试，覆盖默认值、合法角色、非法角色和角色更新场景
5. 如需回滚，先回退 `user` 模块代码与测试，再处理数据库字段兼容或手动清理新增字段

## Open Questions
