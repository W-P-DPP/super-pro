## Context

当前 `general-server` 的 `POST /api/user/loginUser` 会返回 JWT 与用户信息，`login-template` 直接提交明文密码，并且登录成功后没有把认证令牌持久化到本地。现有 JWT 规则已经调整为按路由显式挂载中间件，`user` 模块通过在 `jwtMiddleware` 之前声明匿名路由来保留登录和注册能力，因此本次变更必须延续该边界设计，而不能回退到白名单模式。

这次变更同时跨越后端登录契约、前端认证缓存和登录安全链路，属于典型的跨模块安全变更。设计需要在不扩大页面复杂度的前提下，给出一个前后端都能稳定实现的密码加密方案，并明确 breaking API 的兼容边界。

## Goals / Non-Goals

**Goals:**

- 收缩登录接口响应，只返回建立登录态所需的 token 元数据。
- 在前端登录成功后将认证令牌持久化到本地缓存，供后续请求读取。
- 为登录链路增加“前端加密、后端解密、服务端验密”的应用层加密流程。
- 明确匿名登录相关接口的路由边界，继续遵守按需挂载 JWT 的仓库规范。

**Non-Goals:**

- 不修改注册接口的密码传输方案，本次仅覆盖登录链路。
- 不新增用户资料接口或登录后用户信息回填流程。
- 不改变现有 JWT 中间件的总体策略，也不引入白名单逻辑。
- 不扩展登录页视觉层级，页面仍保持现有最小表单体验。

## Decisions

### 1. 使用匿名公钥接口 + RSA-OAEP 作为登录密码加密方案

后端新增匿名接口 `GET /api/user/getLoginPublicKey`，由 `user` 模块在 `jwtMiddleware` 之前显式挂载。该接口返回 PEM 格式的登录公钥，前端登录时先获取公钥，再使用浏览器 Web Crypto 的 `RSA-OAEP + SHA-256` 对密码进行加密，随后提交 `username` 与 `passwordCiphertext`。后端从受保护配置中读取私钥，先解密出原始密码，再复用现有 `scrypt` 哈希校验逻辑。

这样做的原因是前端需要一个可以公开分发的密钥材料，而私钥必须严格留在服务端。相比把对称密钥写进前端构建产物，公钥下发不会暴露可逆解密能力；相比把密码先做一次前端哈希再提交，非对称加密更符合“后端解密并验证密码”的需求表达，也不会把前端计算结果变成可重放凭证。

备选方案：

- 前端内置固定对称密钥：放弃，因为密钥会随前端代码暴露。
- 前端只做一次哈希：放弃，因为后端无法“解密”，且哈希结果会变成新的静态登录凭证。

### 2. 登录请求契约改为显式密文字段，登录响应契约改为 token-only

后端登录请求 DTO 改为接收 `username` 与 `passwordCiphertext`，不再接受明文 `password`。登录成功后的响应 DTO 仅保留 `token`、`tokenType`、`expiresIn`，移除 `user` 字段。前端登录 API 适配新的请求与响应结构，登录成功后的页面反馈不再依赖接口返回的用户信息。

这样做可以把 breaking change 明确到字段层，避免“password 字段有时是明文、有时是密文”的歧义；同时收缩返回体，满足最小认证数据返回原则。如果后续前端需要展示当前用户资料，应通过独立的受保护接口获取，而不是继续让登录接口承担资料下发职责。

备选方案：

- 保留 `password` 字段名称但改传密文：放弃，因为协议语义模糊，联调和排障成本更高。
- 保留 `user` 字段但前端忽略：放弃，因为接口仍然过度返回，与需求相悖。

### 3. 前端本地缓存使用 localStorage，并限制保存内容为认证会话元数据

`login-template` 新增独立的认证存储封装，在登录成功后将 `token`、`tokenType` 与基于 `expiresIn` 计算出的过期时间持久化到 `localStorage`。前端不再在本地保存登录返回的用户信息，因为登录接口不再提供该数据。后续如有鉴权请求封装，可统一从该存储中读取 bearer token。

之所以选择 `localStorage`，是因为用户明确要求“本地缓存保存 token”，且当前页面已经去掉“记住设备”之类的分支，不再需要在 `localStorage` 和 `sessionStorage` 之间切换。保留 tokenType 与过期时间是为了减少后续请求层和过期判断的重复计算，但本次仍以 token 持久化为主目标。

备选方案：

- `sessionStorage`：放弃，因为不满足“本地缓存保存 token”的默认持久化诉求。
- 只存 token 字符串：未采用为默认实现，因为会丢失显式过期信息，不利于后续扩展。

### 4. 匿名接口边界继续在 user router 内显式声明

`GET /api/user/getLoginPublicKey` 与 `POST /api/user/loginUser` 继续作为匿名接口放在 `userRouter.use(jwtMiddleware)` 之前；现有 `POST /api/user/registerUser` 的匿名能力不在本次需求中变更。`general-server/src/index.ts` 仍然只在业务前缀层挂载 router，不在 `jwtMiddleware` 内维护任何路径白名单。

这样做保持了仓库已经确定的 JWT 设计方向，也避免登录链路新增一处“例外规则”后再次把匿名访问散落回全局中间件。

## Risks / Trade-offs

- [公私钥配置缺失或格式错误] → 在后端启动或首次调用时进行配置校验，并返回受控中文错误，避免把底层加解密异常直接暴露给前端。
- [前端公钥获取失败或浏览器加密失败] → 登录前给出受控错误提示，阻止发送明文密码。
- [localStorage 令牌受 XSS 风险影响] → 本次按用户要求采用本地持久化，但只保存认证元数据，不额外保存用户资料，并保持页面实现最小化。
- [前后端版本不一致导致登录失败] → 作为 breaking change 统一发布前后端；若需回滚，应同时恢复旧的登录请求/响应契约。

## Migration Plan

1. 先在后端补齐登录公钥接口、私钥解密能力和新的登录 DTO/响应 DTO。
2. 再让前端切换到“获取公钥 -> 加密密码 -> 提交密文 -> 本地持久化 token”的新链路。
3. 联调验证通过后统一发布，避免新前端调用旧后端或旧前端调用新后端造成登录不可用。
4. 如需回滚，前后端一并恢复旧契约，避免系统处于半切换状态。
