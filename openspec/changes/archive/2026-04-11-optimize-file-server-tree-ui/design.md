## Context

当前 `file-server` 已经能通过 `GET /api/file/tree`、`POST /api/file/folder`、`POST /api/file/upload` 和 `DELETE /api/file` 完成基础文件管理，但页面结构仍然沿用“左侧树导航 + 右侧主操作区”的组织方式。左侧树只承担目录切换，真正的新增、上传、删除主要落在右侧面板和内容列表中。

这会带来两个直接问题：

- 交互中心不稳定。用户既要看树，又要切到右侧寻找操作入口。
- 页面责任不清晰。左侧是目录树，右侧既是当前目录列表又是主要操作台，导致同一个任务路径被拆成两块。

本次需求希望把 `file-server` 调整为典型的树工作区：左侧树就是主要操作面，右侧内容区暂时不承载业务内容，只保留空白占位，为未来详情视图或预览视图留边界。

## Goals / Non-Goals

**Goals:**

- 将左侧文件树升级为 `file-server` 的主要工作区
- 让选中、上传、新建文件夹、删除文件或文件夹围绕树节点完成
- 让右侧内容区先退化为稳定的空白占位态，而不是继续承担当前目录列表
- 保持现有后端文件服务接口不变，只调整前端布局和交互编排
- 保持与仓库既有前端设计语言一致，继续使用现有 token、主题和字体体系

**Non-Goals:**

- 本次不新增后端接口或变更现有文件服务协议
- 本次不实现右侧文件详情、预览、属性面板或拖拽操作
- 本次不实现树节点右键菜单、批量选择或复杂快捷键体系
- 本次不重做整套主题，只在现有设计体系内重组布局

## Decisions

### 决策 1：左侧树从“导航区”升级为“树工作区”

左侧区域不再只是目录导航，而是承担以下职责：

- 展示完整的目录与文件层级
- 表示当前选中节点
- 在树节点附近提供新增、上传、删除等操作入口
- 作为主要的文件管理操作流起点

选择这种方案，而不是继续保留右侧主操作面板的原因：

- 文件管理天然适合围绕树结构进行就地操作
- 交互路径更短，用户不需要在目录树和操作面板之间来回跳转
- 页面结构更稳定，后续右侧内容区可以单独扩展，不再和基础操作耦合

### 决策 2：树中同时展示文件和文件夹，并区分可操作动作

树不再只显示文件夹，而是显示完整文件系统层级。节点行为分为两类：

- 文件夹节点：可选中、展开/收起、上传到该目录、新建子文件夹、删除目录
- 文件节点：可选中、删除文件

选择“完整树”而不是“仅目录树 + 右侧文件列表”的原因：

- 用户要求文件增删查都在树中操作
- 如果树里不显示文件，删除文件仍需要依赖右侧列表，目标无法达成
- 完整树能把文件系统的层级关系表达得更直接

### 决策 3：右侧内容区保留为空白占位，而不是继续展示当前目录列表

右侧内容区本次不继续显示当前目录内容表格，而是改为一个明确的空白占位区，例如：

- 提示当前已选中的节点
- 说明“内容区暂未启用，后续可承载预览/详情/编辑能力”

选择占位而不是保留现有列表的原因：

- 保留列表会形成新的“第二操作中心”
- 本次目标是收束交互，不是继续扩展双栏内容
- 留空可以把未来扩展边界表达得更清楚

### 决策 4：树节点操作采用轻量内联入口，而不是大面积工具条

树节点的主要操作建议采用以下方式组合：

- 顶部保留少量全局操作，如刷新和主题切换
- 节点选中后在节点行内显示轻量操作按钮，或在选中节点下方显示紧邻的局部操作栏

选择轻量内联操作，而不是重新做一个显性的左侧大工具箱，原因是：

- 更符合“就地操作”心智
- 不会压缩树的可读区域
- 与现有克制、专业的设计约束更一致

### 决策 5：视觉上继续沿用现有主题 token，不引入新视觉体系

本次变更继续使用：

- 现有紫灰系 `primary/secondary/accent`
- 现有 `0.5rem` 基础圆角体系
- 现有轻量阴影和边框表达
- `Geist Variable` 与中文系统字体回退

树工作区的视觉重点通过层级、选中态、局部按钮显隐和信息密度来建立，而不是通过新颜色体系或大面积装饰实现。

## Risks / Trade-offs

- [风险] 完整文件树在深层目录下可能显得信息密度过高
  -> Mitigation：通过缩进、节点图标、选中态和悬停显隐操作控制噪音，避免一次性暴露过多强交互元素

- [风险] 把操作全部收回树中后，移动端可点击区域可能变紧
  -> Mitigation：移动端改为单列树工作区，节点操作入口使用更明确的触控区域或局部折叠动作

- [风险] 用户可能误以为右侧空白区是未加载成功
  -> Mitigation：在空白区提供明确的占位文案，说明当前区域预留给后续详情能力

- [风险] 树中展示文件后，节点数量会明显增多
  -> Mitigation：`v1` 先接受完整树展示，后续若规模变大，再考虑懒展开或按目录增量加载

## Migration Plan

1. 先调整 `file-server` 的页面布局，把左侧区域提升为主要工作区
2. 重构树渲染逻辑，让树同时展示文件和文件夹
3. 将上传、新建文件夹、删除等动作从右侧面板迁移到树工作区
4. 将右侧内容区改为空白占位态，并移除当前目录表格式主操作内容
5. 在浅色和深色主题下验证树的层级、选中和悬停表现一致

回滚策略：

- 如果树工作区交互效果不理想，可回滚 `file-server` 的页面布局和组件拆分
- 由于本次不涉及后端协议变更，回滚不会影响文件服务 API

## Open Questions

- 节点操作入口最终采用“行内 hover 显示”还是“选中节点后固定显示局部工具条”；本次实现时可先选更稳妥的一种
- 文件节点是否需要单独的选中态文案区；本次可先只保证树中可见和可删除
