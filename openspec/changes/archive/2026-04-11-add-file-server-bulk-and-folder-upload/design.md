## Context

当前 `file-server` 的上传链路是“前端单文件输入 + 后端 `multer.single('file')` + repository 单文件落盘”。这条链路已经满足基础上传，但它把上传能力限制在“一个目标目录一次一个文件”，无法覆盖两个高频场景：

- 用户希望一次把若干文件上传到当前目录
- 用户希望把本地文件夹整体上传，并保留该文件夹内部的层级结构

这次变更同时涉及后端 `file` 模块和前端 `file-server` 树工作区，因此需要先把请求契约、冲突处理规则和前端交互方式定清楚，避免前后端各自扩展出不兼容的上传模型。

关键约束：

- 继续复用现有 `/api/file/upload` 路由，不新增新的业务域
- 继续把仓库根 `file` 目录作为唯一可写业务根目录
- 继续拒绝覆盖同名目标
- 需要兼容当前单文件上传调用，避免已有链路立刻失效

## Goals / Non-Goals

**Goals:**

- 让 `/api/file/upload` 支持单文件、批量文件和文件夹上传三种模式
- 让系统在文件夹上传时按客户端相对路径重建目录结构
- 让批量/文件夹上传沿用现有路径保护和冲突保护
- 让单次上传请求采用整批校验、整批写入，避免部分成功部分失败
- 让 `file-server` 从树工作区直接触发批量文件上传和文件夹上传

**Non-Goals:**

- 本次不实现断点续传、上传进度条、并发分片或超大文件优化
- 本次不实现拖拽上传和右键菜单扩展
- 本次不新增独立的批量上传任务记录、回滚任务或后台异步队列
- 本次不改变 `file` 根目录和 `rubbish` 目录的业务边界

## Decisions

### 决策 1：继续复用 `/api/file/upload`，扩展为“多文件 + 可选相对路径”契约

后端继续使用现有上传路由 `/api/file/upload`，但 multipart 契约从单文件扩展为：

- `targetPath`: 目标目录
- `files`: 重复文件字段，用于批量上传和文件夹上传
- `relativePaths`: 与 `files` 按索引对齐的可选相对路径

同时为兼容现有单文件客户端，后端仍接受旧的 `file` 字段；如果请求只有单个 `file` 且没有 `files`，服务端会自动把它视作一个单元素上传批次。

这样做的原因：

- 避免为“批量上传”和“文件夹上传”拆出多个几乎相同的后端接口
- 让单文件、批量文件、文件夹上传共用一套校验和落盘逻辑
- 保持旧客户端在不升级时仍然可用

备选方案：

- 新增 `/api/file/uploadBatch` 和 `/api/file/uploadFolder`：语义更直白，但会复制大部分校验和落盘逻辑，维护成本更高

### 决策 2：以后端显式 `relativePaths` 作为文件夹重建依据

前端在批量/文件夹上传时，将每个文件都附带一个与其对应的 `relativePaths` 条目。普通批量文件上传时，该值为文件名；文件夹上传时，该值取自浏览器的 `webkitRelativePath`。后端不依赖 multipart 原始文件名字段来推断目录层级，而是用 `relativePaths` 决定目标子路径，再结合文件名标准化逻辑决定最终文件名。

这样做的原因：

- `FormData` 自身不会自动保留目录结构，必须显式传递相对路径
- 把目录结构信息和文件名字节解码问题分开处理，职责更清晰
- 便于在服务端统一做路径清洗和越界保护

备选方案：

- 完全依赖 `originalname` 推断层级：无法表达文件夹结构，也无法稳定兼容不同浏览器

### 决策 3：批量上传采用整批预检，任一异常时整批拒绝

服务端在写入任何文件前，先完成整批预检：

- 解析每个文件的最终相对目标路径
- 验证所有目标都位于 `file` 根目录内
- 验证批次内部没有重复目标
- 验证目标路径当前不存在冲突文件或文件夹

只有整批预检通过后，才会创建缺失的中间目录并写入所有文件。若任意一项失败，则整批拒绝，且不写入任何文件。

这样做的原因：

- 批量/文件夹上传的预期是“一次导入一组内容”，不应出现半成功状态
- 失败时更容易向前端提供明确反馈，前端只需提示用户修正冲突后重试

备选方案：

- 允许部分成功并返回逐项结果：实现更复杂，用户也需要自行处理残留状态

### 决策 4：文件夹上传允许在目标目录下自动创建缺失的中间目录

当前单文件上传要求 `targetPath` 必须存在；这一点保持不变。但在文件夹上传内部，若 `targetPath` 已存在，则由 `relativePaths` 推导出的中间子目录可以自动创建，例如上传 `images/icons/logo.svg` 到 `/assets` 时，服务端会自动创建 `/assets/images` 和 `/assets/images/icons`。

这样做的原因：

- 文件夹上传本质上就是重建子目录树，如果仍要求每一级手工预创建，功能价值会大幅下降
- 这类自动创建只发生在已确认位于 `targetPath` 下的中间目录，不会放宽根目录边界

### 决策 5：前端树工作区提供两个独立上传入口

`file-server` 前端在文件夹节点上提供两个上传入口：

- 批量文件上传
- 文件夹上传

两者都仍然以当前树节点对应目录作为 `targetPath`。前端使用两个隐藏输入框管理两种选择模式，并在提交后统一刷新树、保留目标目录焦点、显示中文成功/失败反馈。

这样做的原因：

- 批量文件上传和文件夹上传的浏览器选择器不同，分成两个入口更直观
- 不需要引入复杂菜单或弹层，就能把能力落在现有树工作区上

## Risks / Trade-offs

- [风险] 批量上传请求失败时只能给出整批失败结果，用户无法直接知道每个文件的详细状态
  -> Mitigation: 后端优先返回最关键的冲突或路径错误信息，前端明确提示整批未写入

- [风险] 文件夹上传依赖浏览器提供的 `webkitRelativePath`，浏览器兼容性不是完全统一
  -> Mitigation: 前端将该能力限定在支持目录选择的浏览器环境；普通多文件上传仍然可用

- [风险] 一次上传过多文件时，内存型 multipart 处理会放大内存占用
  -> Mitigation: 本次先沿用现有 `multer.memoryStorage()` 实现；如后续出现大体量上传需求，再单独评估流式落盘

## Migration Plan

1. 扩展 `file` 模块 DTO、controller、service、repository，使其支持上传批次和相对路径
2. 增加整批预检、内部冲突校验和中间目录创建
3. 调整 `file-server` 树工作区上传入口与表单构造逻辑
4. 增加后端单元/集成测试，覆盖批量与文件夹上传
5. 运行后端测试与前端构建验证

回滚策略：

- 若新上传链路存在问题，可回滚到单文件上传实现
- 因为继续复用原路由且兼容旧 `file` 字段，回滚不会影响其他业务域

## Open Questions

- 当前没有阻塞实现的开放问题；如后续用户需要上传进度、拖拽上传或超大文件优化，应另起 change 处理
