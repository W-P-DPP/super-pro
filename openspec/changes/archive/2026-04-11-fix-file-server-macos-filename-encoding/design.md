## Context

当前 `general-server` 的文件上传接口通过 `multer` 接收 multipart 表单，并直接把 `req.file.originalname` 透传到 `fileService.uploadFile()`。现有实现默认这个字段已经是可直接落盘的正确文件名，但 macOS 浏览器上传中文文件名时，multipart 头中的原始字节可能被按错误字符集解释，导致服务端看到的 `originalname` 已经是乱码。

这个问题集中在 `POST /api/file/upload` 链路，但会连带影响三个关键行为：

- 上传成功后的实际落盘名称
- 同名冲突检测使用的文件名
- 接口响应里返回给前端展示的文件名

约束也很明确：

- 不新增接口，不改变 `file-server` 的调用方式
- 不修改 repository 的根路径保护和落盘职责
- 需要兼容现有正常 UTF-8 文件名，不能为了修复 macOS 场景破坏已有上传

## Goals / Non-Goals

**Goals:**

- 让 `POST /api/file/upload` 在常见跨平台浏览器上传场景下尽量保留用户原始文件名
- 将文件名解码、清洗和合法性校验收敛为单一入口，避免 controller、service、repository 各自处理一部分
- 让冲突校验、最终写入和响应回包都基于同一份已规范化文件名
- 为 macOS 中文文件名上传补齐单元和集成测试

**Non-Goals:**

- 本次不新增批量上传、多文件拖拽或下载文件名编码处理
- 本次不引入数据库记录“原始文件名”与“存储文件名”的双写模型
- 本次不尝试覆盖所有历史浏览器和所有 multipart 编码变体，只解决当前仓库可复现的 macOS 中文文件名乱码问题

## Decisions

### 决策 1：在文件模块内部新增统一的上传文件名标准化步骤

文件名兼容逻辑放在 `file` 模块内部完成，而不是散落在前端或 repository 层。具体上，controller 仍然负责取出 `req.file`，service 在校验上传文件时调用统一的文件名标准化逻辑，得到最终可用的 `originalname`。

这样做的原因：

- controller 只负责 HTTP 输入映射，不应该承载字符集兼容规则
- repository 应该只处理已经合法的业务输入，不应承担 multipart 解析副作用
- service 层已经负责上传参数校验，把文件名标准化并入这里，能保证校验、冲突检测和落盘使用同一结果

备选方案：

- 在前端上传前重新编码文件名：不可取，浏览器和平台差异仍会泄漏到客户端，而且无法覆盖非前端调用者
- 在 repository 落盘前再尝试解码：过晚，会让上游校验与最终写入依据不一致

### 决策 2：采用“优先保留原值，疑似乱码时再做兼容解码”的保守策略

文件名处理采用保守策略：默认保留 `multer` 给出的 `originalname`；只有在判断该值明显呈现典型乱码特征时，才尝试用兼容方式重新解释原始字节并恢复 UTF-8 文件名。若兼容解码失败，则回退到原值，再继续现有 basename 与合法性校验。

这样做的原因：

- 正常 UTF-8 文件名已经工作，不应该被无条件二次转换
- macOS 中文文件名乱码通常表现为“字节正确、解释错误”，适合在服务端做一次有限纠偏
- 回退到原值可以避免因为激进解码导致英文名、符号名或已正确的 Unicode 名称被二次破坏

备选方案：

- 对所有文件名统一执行 `latin1 -> utf8` 重解码：风险过高，可能破坏本来正确的文件名
- 完全依赖客户端提供额外字段传递文件名：需要改协议，不符合本次目标

### 决策 3：冲突检测和响应回包都以规范化后的文件名为准

一旦得到规范化后的文件名，后续链路必须全部复用这份名称，包括：

- `validateUploadedFile()` 的合法性校验
- repository 的目标路径拼接与冲突检测
- 成功响应中返回的 `name` 和 `relativePath`

这样做的原因：

- 如果校验和落盘用不同名称，会出现“校验通过但写入冲突”或“前端看到的名称和磁盘不一致”
- 测试可以围绕最终规范化名称建立稳定断言

备选方案：

- 仅在响应阶段替换显示名称：不可取，磁盘名称和冲突校验仍然错误

## Risks / Trade-offs

- [风险] 乱码识别规则过松，可能把本来合法但少见的文件名误判为需要重解码
  -> Mitigation: 采用保守判定和失败回退，只在典型乱码特征下尝试兼容解码，并为英文名、普通中文名分别补测试

- [风险] 只修复 `file` 模块，未来其他上传模块可能重复踩坑
  -> Mitigation: 先在 `file` 模块验证规则；若后续 `siteMenu` 等模块出现同类问题，再抽成共享上传文件名工具

- [风险] 已经以乱码文件名落盘的历史文件不会自动修复
  -> Mitigation: 本次仅修复新上传行为，不做历史文件迁移；如需处理存量数据，另起 change

## Migration Plan

1. 在 `file` 上传链路中补充统一的文件名标准化逻辑
2. 保持现有 API 协议不变，直接替换服务端对上传文件名的解释方式
3. 增加单元测试覆盖乱码解码和冲突校验，增加集成测试覆盖上传后落盘名称
4. 合并后直接发布，无需额外数据迁移

回滚策略：

- 若兼容解码引入异常，可回滚 `file` 模块中的文件名标准化逻辑，恢复为直接使用 `req.file.originalname`
- 由于没有协议和数据结构变更，回滚不会影响现有前端调用
