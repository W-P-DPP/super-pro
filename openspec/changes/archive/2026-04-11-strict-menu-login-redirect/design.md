## Context

`frontend-template` 当前已经能从后端读取菜单树，并在 `HomePage` 中用卡片形式展示每个工具入口，但点击逻辑只区分“外部链接直接打开”和“站内路径暂未接入”，没有消费菜单项的 `strict` 字段。与此同时，`login-template` 登录成功后只保存 token 并展示成功反馈，不会读取 URL 中的重定向目标，因此即使未来从别的项目跳到登录页，也无法回到原始地址。

这次变更跨越 `frontend-template` 和 `login-template` 两个独立前端包，且涉及环境变量、跨项目跳转和登录后导航恢复，属于需要先明确跳转契约的跨模块前端设计变更。视觉层面仍保持现有克制、单卡片、中文文案风格，不引入额外页面结构。

## Goals / Non-Goals

**Goals:**

- 让 `frontend-template` 在点击 `strict === true` 的菜单卡片时，统一跳到配置化的登录页地址，而不是直接进入原始目标。
- 约定 strict 菜单跳转时附带 `redirect` 参数，参数值为被点击卡片的原始跳转地址。
- 让 `login-template` 在登录成功后读取 `redirect` 参数并跳回原始目标地址。
- 将登录页地址抽成环境变量，便于后续替换部署域名或端口。

**Non-Goals:**

- 不修改后端菜单接口，也不改变 `strict` 字段的来源和含义。
- 不改变非 strict 菜单项现有的点击行为。
- 不新增登录页视觉元素或新的表单字段。
- 不引入新的全局鉴权框架或单点登录机制。

## Decisions

### 1. `frontend-template` 使用独立环境变量管理登录页地址

在 `frontend-template` 中新增如 `VITE_STRICT_MENU_LOGIN_URL` 的环境变量，默认值为 `http://www.zwpsite.icu:8082/login`。严格菜单卡片被点击时，不在组件内写死登录页地址，而是统一通过该配置生成最终登录跳转 URL。

这样做的原因是登录页地址已经是跨项目、跨域名的外部地址，后续很可能随着部署环境变化而调整。把它直接散落在卡片点击逻辑里会提高维护成本，也不利于多环境切换。

备选方案：

- 在组件中写死登录页 URL：放弃，因为后续修改成本高且容易遗漏。
- 通过后端返回登录地址：放弃，因为当前需求只涉及前端跳转策略，不需要为此扩展接口。

### 2. strict 菜单跳转统一使用 `redirect` 查询参数承载原始目标地址

`frontend-template` 在点击 strict 卡片时，使用 `redirect` 作为查询参数名，并将卡片原始目标地址完整编码后拼接到登录页 URL，例如 `http://www.zwpsite.icu:8082/login?redirect=<encoded-target>`。这里的目标地址直接使用被点击卡片的 `path` 字段值，不在 `frontend-template` 侧做额外协议改写。

这样做可以让登录页保持最小上下文，只需要消费一个明确的重定向参数。参数名保持通用、语义直接，也便于后续其它前端入口复用同一套规则。

备选方案：

- 使用 hash 片段携带重定向地址：放弃，因为登录页读取和与其它查询参数共存都不如标准 query 参数直接。
- 将目标地址拆成多个字段：放弃，因为需求只要求恢复单一跳转目标，没有必要复杂化协议。

### 3. `login-template` 在登录成功后优先执行外部重定向，无参数时维持当前成功反馈

`login-template` 在登录成功并保存 token 后，从当前 URL 的查询参数中读取 `redirect`。若参数存在且非空，则调用 `window.location.assign(redirect)` 执行跳转；若参数不存在，则继续保留现有“登录成功，已保存登录凭证”的本地反馈。为避免把明显无效值当成跳转目标，设计上要求至少对空值做拦截；实现时可进一步限制为 `http://`、`https://` 或以 `/` 开头的地址。

这样做可以让登录页既支持从 strict 菜单入口回跳，也不破坏它作为独立登录页的现有使用方式。

备选方案：

- 登录成功后始终强制跳转首页：放弃，因为会丢失原始目标地址。
- 一律依赖 React Router 导航：放弃，因为登录页当前没有路由层，且重定向目标可能是外部地址。

### 4. strict 规则仅作用于卡片点击入口，不改变目录侧边栏滚动行为

`frontend-template` 左侧目录当前只负责滚动到页面 section，而不是直接进入工具目标地址。这次 strict 规则仅作用于工具卡片点击跳转，不扩展到目录滚动按钮或其它非目标跳转交互。

这样做可以把需求严格限制在“点击卡片进入工具”这条主路径上，避免把侧边栏目录从页内导航变成外部路由入口，破坏现有信息架构。

## Risks / Trade-offs

- [登录页环境变量配置错误] → 提案要求提供默认值，并把 URL 生成集中在单一工具函数中，降低分散配置风险。
- [redirect 参数为空或格式异常] → 登录页在消费参数时先做基础校验；无效时回退到本地成功反馈，不执行跳转。
- [strict 菜单目标地址本身不可访问] → 本次只保证登录链路和参数传递，不负责校验目标系统可用性。
- [跨项目重定向导致调试链路变长] → 通过明确的 `redirect` 约定和环境变量减少排查面，并保留非 strict 入口原有直达行为。

## Migration Plan

1. 先在 `frontend-template` 增加 strict 菜单跳转工具函数和登录页地址环境变量。
2. 再在 `login-template` 接入登录成功后的 `redirect` 参数消费逻辑。
3. 联调验证 strict 卡片、非 strict 卡片、无 `redirect` 登录三条路径。
4. 如需回滚，两个前端包分别移除 strict 跳登录和登录成功回跳逻辑即可，不涉及后端回滚。

## Open Questions
