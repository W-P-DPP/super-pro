## Context

当前后端应用入口 `general-server/app.ts` 已经完成了通用中间件、统一响应中间件和 `/api` 业务路由挂载，但没有对 `general-server/public` 目录做静态资源托管。仓库中已经存在 `public/icons/*.svg`、`public/icons/*.ico`、`public/1.txt` 等文件，说明静态资源目录已经投入使用，只是服务链路尚未打通。

由于现有业务接口全部挂在 `/api` 下，并且会经过 JWT 与业务日志中间件，因此静态资源服务需要与 `/api` 业务域明确隔离，避免静态文件访问被鉴权、日志包装或业务路由影响。

## Goals / Non-Goals

**Goals:**

- 启动后端时自动托管 `general-server/public` 目录
- 让 `public` 下的文件可以通过 HTTP 直接访问，例如 `/icons/pin.svg`、`/1.txt`
- 保持 `/api` 业务路由、中文返回和现有中间件行为不变
- 补充静态资源访问成功和缺失资源返回 404 的验证

**Non-Goals:**

- 不新增新的 `/api` 业务接口
- 不新增数据库、DTO、repository、service 之类业务模块文件
- 不引入额外静态资源服务器或 CDN
- 不在本次加入上传、缓存策略配置或目录浏览能力

## Decisions

### 决策 1：在应用入口层直接挂载 `express.static(publicPath)`

静态资源服务属于基础设施能力，而不是业务域能力，因此应直接在 `app.ts` 入口层挂载，而不是拆进某个业务模块。实现时使用 Express 自带的静态资源中间件，托管 `general-server/public` 对应的绝对目录。

选择这一方案而不是新建业务 router 的原因：

- 静态资源不是领域行为，不应混入 `siteMenu`、`user` 等业务模块
- 入口层最适合声明“请求路径 -> 资源目录”的基础设施映射
- 不会破坏用户要求的业务模块六层结构，因为这里不是新增业务功能模块

### 决策 2：静态资源挂载在 `/api` 之前，并直接暴露为根路径资源

实现时将 `public` 目录挂载在应用根路径下，而不是 `/api/public` 或其他业务前缀下。这样 `public/icons/pin.svg` 可直接访问为 `/icons/pin.svg`，`public/1.txt` 可直接访问为 `/1.txt`。

选择根路径挂载而不是 `/public` 前缀的原因：

- 仓库里的现有图标路径已经大量使用 `/icons/...`
- 可以减少前端和菜单数据中的资源地址改动
- 资源访问不会被 `/api` 下的 JWT 和业务中间件拦截

### 决策 3：缺失资源保持框架默认 404，不包装业务响应体

静态资源访问属于标准 HTTP 文件服务，不应走业务 `sendSuccess/sendFail` 包装。缺失文件时保持静态资源中间件和默认 404 语义即可。

选择这一方案而不是返回统一 JSON 的原因：

- 静态资源请求面向浏览器和文件消费方，应优先遵循标准静态文件行为
- 能避免把基础文件服务混成业务接口契约
- 更符合 Nginx、浏览器缓存和资源加载的常见预期

## Risks / Trade-offs

- [风险] 根路径挂载后，若静态文件名与现有非 `/api` 路由冲突，可能产生覆盖
  → Mitigation：当前应用除 `/` 首页探活外没有复杂根路径路由，且静态资源中间件可放在合适顺序避免误伤

- [风险] 静态资源请求不会走统一中文响应包装
  → Mitigation：这是刻意保留的标准文件服务行为，仅对业务接口继续要求统一中文返回

- [风险] 后续若资源量增大，可能需要缓存头或 CDN
  → Mitigation：本次先完成本地静态托管，后续再按需要扩展缓存策略

## Migration Plan

1. 在 `app.ts` 中定位 `public` 目录绝对路径并挂载静态资源中间件
2. 确保静态资源挂载顺序在 `/api` 业务路由之前
3. 补充集成测试，验证已存在文件可访问、缺失文件返回 404
4. 回归验证现有 `/api` 业务接口不受影响

回滚策略：

- 若静态资源挂载引发路径冲突，可回滚 `app.ts` 中的静态资源中间件改动
- 回滚后 `public` 目录恢复为仅仓库存储，不再对外服务
