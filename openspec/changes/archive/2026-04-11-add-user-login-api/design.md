## Context

当前 `general-server` 已具备完整的 `user` 六层模块和独立的 JWT 中间件，但两部分尚未形成完整登录链路：

- `user` 模块只有 CRUD，没有登录 DTO、登录服务方法和登录路由
- `jwtMiddleware` 只负责校验 bearer token，并提供 `generateToken`，没有登录放行策略
- `app.ts` 当前把 `/api` 下所有业务路由统一挂在 `jwtMiddleware` 之后；当 `JWT_ENABLED=true` 时，登录接口如果直接加到 `userRouter` 会被中间件先拦截
- `sys_user` 当前只有基础资料字段，没有可用于认证的密码字段

因此，这次变更不仅要新增登录接口，还要补齐“凭证存储 + 登录校验 + token 签发 + 登录路径免鉴权”这条完整链路，并继续遵循仓库现有的动作式路由、中文响应、六层分离和测试覆盖约束。

## Goals / Non-Goals

**Goals:**

- 在 `user` 模块内新增 `POST /api/user/loginUser` 登录接口
- 为 `sys_user` 补充可用于认证的密码哈希字段，并保持敏感字段不对外返回
- 在 service 层完成用户名、密码、用户状态校验，并在成功后签发 JWT
- 在 JWT 开启时允许登录接口匿名访问，其余现有受保护接口仍按原规则校验 token
- 补充与登录接口匹配的单元测试和集成测试

**Non-Goals:**

- 不在本次变更中实现注册、登出、刷新 token、角色权限或验证码能力
- 不接入新的外部认证服务，不实现短信、邮箱或第三方登录
- 不在本次变更中重构全局响应结构或替换现有 JWT 机制

## Decisions

### 决策 1：登录能力仍归属 `user` 业务域，不新拆 `auth` 模块

登录接口新增在现有 `general-server/src/user/` 六层文件内，由 `user.router.ts` 暴露 `POST /loginUser`，并由 `user.controller.ts`、`user.service.ts`、`user.repository.ts`、`user.dto.ts` 分层承载。

这样做优于新建独立 `auth` 模块的原因：

- 当前仓库还没有独立的认证域模型，登录只依赖 `sys_user` 数据，直接放在 `user` 域成本更低
- 能复用既有 `user` 模块的实体、仓储和测试结构，避免过早拆分
- 与用户当前请求“在 use/user 功能中新增登录接口”保持一致

备选方案：

- 新建 `auth` 模块，单独管理登录与 token：边界更纯，但对当前仓库是过度设计，且会引入跨模块编排

### 决策 2：登录采用“用户名 + 密码”模型，并在 `sys_user` 增加密码哈希字段

登录请求使用用户名和密码；`sys_user` 新增密码哈希字段，仓库内只存哈希值，不回传明文，也不在普通用户查询接口中暴露。

这样做优于“直接用现有资料字段作为登录凭证”的原因：

- 现有 `username`、`nickname`、`email`、`phone` 都属于资料字段，不能替代密码
- 为后续注册、修改密码、重置密码等能力预留了明确扩展位
- 与后端守卫中的安全要求一致，避免把敏感信息直接写入响应或日志

备选方案：

- 存明文密码：实现简单，但安全性不可接受
- 引入外部加密依赖：可行，但当前只需要基础登录，优先使用 Node 内置密码哈希能力降低依赖面

### 决策 3：密码哈希优先使用 Node 内置加密能力，不新增第三方依赖

实现层建议使用 Node 内置 `crypto` 能力生成和校验密码哈希，例如 `scrypt` 或 `pbkdf2`，并将密码处理封装在 `user.service.ts` 内部辅助函数或专用工具函数中。

这样做优于引入 `bcrypt` 一类新依赖的原因：

- 当前仓库未引入专门密码库，内置能力足以覆盖本次基础登录需求
- 减少依赖升级、原生编译和跨环境安装成本
- 对 OpenSpec 当前目标足够，后续若要统一认证标准，可再增量替换

备选方案：

- 使用 `bcrypt`：社区成熟，但会新增依赖和环境复杂度

### 决策 4：登录失败对外统一为中文受控错误，凭证错误不区分“用户不存在”和“密码错误”

当用户名不存在或密码不匹配时，对外统一返回受控中文错误，例如“用户名或密码错误”；当用户状态为禁用时，单独返回“用户已停用，无法登录”之类的业务错误。

这样做的原因：

- 避免通过错误文案枚举系统中的用户名
- 与安全设计一致，减少认证信息泄露
- 仍然保留足够清晰的业务反馈，便于前端提示

备选方案：

- 分别返回“用户不存在”和“密码错误”：调试更直观，但会泄露账户存在性

### 决策 5：通过 JWT 放行白名单让登录接口匿名访问

`app.ts` 当前在 `/api` 层统一挂载 `jwtMiddleware`。本次实现时需要对登录接口增加匿名放行策略，例如在 `jwtMiddleware` 内或路由接入层基于 `req.path` + `req.method` 判断 `POST /api/user/loginUser` 为白名单接口。

这样做优于把登录接口挂到 `/api` 之外的原因：

- 保持所有业务接口仍在 `/api` 命名空间下，风格一致
- 不需要为单一登录接口额外拆分新的入口路由
- 变更范围集中，可最小化影响现有路由组织

备选方案：

- 把登录接口挂到 `/login` 或 `/auth` 根路径：可以绕开 JWT，但会破坏现有统一前缀风格

### 决策 6：登录成功响应同时返回 token 和脱敏后的用户信息

登录成功时返回结构应至少包含：

- `token`
- `tokenType`，固定为 `Bearer`
- `expiresIn`
- `user`，只包含对外可见的用户基础信息，不包含密码哈希

这样做的原因：

- 前端登录后可直接持有 token 和当前登录用户信息，减少额外查询
- 响应契约清晰，便于后续扩展刷新时间或权限摘要

## Risks / Trade-offs

- [风险] 为 `sys_user` 增加密码哈希字段后，既有种子数据和测试数据需要同步补齐
  → Mitigation：在测试种子和初始化数据中增加固定测试密码，确保集成测试稳定

- [风险] JWT 放行策略如果写得过宽，可能意外放开非登录接口
  → Mitigation：放行条件必须同时绑定 HTTP 方法和精确路径，只放行 `POST /api/user/loginUser`

- [风险] 使用 Node 内置加密能力时，不同实现细节可能影响后续迁移
  → Mitigation：将哈希与校验逻辑集中封装，避免散落在 controller 或 repository

- [风险] 当前仓库使用 `synchronize: true`，新增字段会直接影响本地数据库结构
  → Mitigation：在实现前确认测试与开发环境种子兼容，必要时补充默认数据初始化与回滚说明

## Migration Plan

1. 扩展 `user.entity.ts` 与测试种子，为 `sys_user` 增加密码哈希字段
2. 扩展 `user.dto.ts`、`user.repository.ts`、`user.service.ts`、`user.controller.ts`、`user.router.ts`，新增登录契约、查询方法和登录接口
3. 调整 `jwtMiddleware` 或 `/api` 挂载策略，仅对白名单登录接口放行匿名请求
4. 新增 `user` 登录单元测试和集成测试，覆盖成功、参数错误、凭证错误、停用用户和 JWT 开启下的放行行为
5. 执行相关测试后再进入实现阶段

回滚策略：

- 回滚 `general-server/src/user/*` 中登录相关改动和 `jwtMiddleware` 白名单逻辑
- 如新增字段已同步到数据库但变更未启用，可保留兼容字段，或在受控环境中手动回退表结构

## Open Questions

- 既有 `createUser` 接口是否要在本次同步支持传入初始密码，还是先由测试种子和数据库默认值满足登录能力
- token 中最小 payload 是否只保留 `userId`，还是补充 `username` 以便后续审计
